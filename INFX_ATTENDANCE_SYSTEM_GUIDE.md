# 출퇴근 SaaS 시스템 설계 문서

## 1. 개요

### 1.1 목적

- 프로젝트 타임로그 웹서비스에 출퇴근 기능 추가
- 출근/퇴근 시간 기록 및 근무시간 자동 계산
- 정정 신청 및 승인 워크플로우 지원

### 1.2 주요 기능

- 출근/퇴근 버튼으로 시간 기록
- 일별/월별 근무시간 계산 (야근 포함)
- 정정 신청 → 팀장 승인/반려 워크플로우
- 모든 이력 보존 (완전 불변 이벤트 로그)
- 데이터 무결성 검증 시스템

## 2. DB 설계

### 2.1 설계 방식

- **완전 불변 이벤트 로그 방식** 채택
- 엔티티 1개로 모든 기록 관리
- **레코드는 생성만 하고, 절대 수정하지 않음**
- 출퇴근/정정신청/승인/반려를 각각 별도 레코드로 기록
- 2개 필드 조합(`sg_type` + `sg_status`)으로 레코드 종류 구분

### 2.2 엔티티 정보

| 항목 | 값 |
|------|-----|
| Entity Type | CustomNonProjectEntity10 |
| Display Name | 출퇴근 기록 |

### 2.3 필드 정의

| 필드명 | Display Name | 타입 | 필수 | 설명 |

```
{이름}_{YYYYMMDD}_{type}_{status}_{HHMMSS}
```

예시:
- `홍길동_20260305_출근_정상_140530` — 확정된 출근

## 3. 이벤트 흐름

### 3.1 기본 출퇴근

```
[출근 버튼 클릭]
└─ 레코드 생성: sg_type=출근, sg_status=정상
   - sg_time = 출근 시간

[퇴근 버튼 클릭]
└─ 레코드 생성: sg_type=퇴근, sg_status=정상
   - sg_time = 퇴근 시간
```

### 3.2 출근 정정 신청 → 승인

```
A0: 출근 (09:30)
 │
└─ R1: 출근 정정 신청 (08:00으로 변경 신청)
    │  - sg_status = 정정신청
     │  - sg_time = 08:00
     │  - sg_parent = A0
     │  - sg_reason = "지하철 지연으로 늦게 기록됨"
     │
     └─ D1: 출근 정정 승인
         │  - sg_status = 승인
         │  - sg_parent = R1
         │  - sg_user = 신청자
         │  - sg_approver = 팀장
         │
         └─ A1: 출근 (08:00) — 새 확정 레코드
              - sg_status = 확정
              - sg_time = 08:00
              - sg_parent = A0
```

### 3.3 출근 정정 신청 → 반려 → 재신청 → 승인/포기

```
A0: 출근 (09:30)
 │
├─ R1: 출근 정정 신청 (08:00)
 │   │  - sg_time = 08:00
 │   │  - sg_parent = A0
 │   │
 │   └─ D1: 출근 정정 반려
 │        - sg_status = 반려
 │        - sg_parent = R1
 │        - sg_user = 신청자
 │        - sg_approver = 팀장
 │        - sg_reason = "증빙 부족"
 │
└─ R2: 출근 정정 신청 (08:30) — 재신청
     │  - sg_time = 08:30
     │  - sg_parent = A0
     │  - sg_reason = "증빙 첨부함"
     │
     └─ D2: 출근 정정 승인
         │  - sg_parent = R2
         │
         └─ A1: 출근 (08:30) — 새 확정 레코드
              - sg_status = 확정
              - sg_time = 08:30
              - sg_parent = A0
```

반려 후 더 이상 진행하지 않으면 `출근 / 포기`로 종료한다.

```
R1: 출근 정정 신청
 │
 └─ D1: 출근 정정 반려
     │  - sg_status = 반려
     │
     └─ X1: 출근 정정 포기
         - sg_status = 포기
         - sg_parent = D1
```

### 3.4 현재 상태 판단 로직

특정 출근(A0)의 정정 상태를 확인하려면:

1. A0를 `sg_parent`로 참조하는 레코드들 조회
2. 가장 최근(`created_at` 기준) 레코드의 `sg_status` 확인

```python
def get_attendance_status(original_attendance):
    """출퇴근 레코드의 현재 상태를 반환한다."""
    # 관련 레코드 조회 (최신순)
    related = sg.find(
        'CustomNonProjectEntity10',
        [['sg_parent', 'is', original_attendance]],
        ['sg_status', 'sg_type', 'created_at'],
        order=[{'field_name': 'created_at', 'direction': 'desc'}]
    )
    
    if not related:
        return '정상'  # 정정 신청 없음
    
    latest = related[0]
    status = latest.get('sg_status')
    
    if status == '확정':
        return '확정'  # 새 확정 레코드가 생성됨 (승인 완료)
    elif status == '정정신청':
        return '대기'  # 승인 대기 중
    elif status == '승인':
        return '승인됨'  # 승인 처리됨 (새 확정 레코드 생성 대기)
    elif status == '반려':
        return '반려됨'  # 반려됨 (재요청 가능)
    elif status == '포기':
        return '포기됨'  # 반려 후 포기
    
    return '정상'
```

## 4. 비즈니스 규칙

### 4.1 근무시간 및 야근 계산

#### 정식 업무 시작 시간

- 정식 업무 시작 시간: **오전 10시 (10:00:00)**
- 의무 근무시간: **9시간** (8시간 근무 + 1시간 점심)

#### 유효 출근 시간 (Effective Start)

출근 시간이 정식 업무 시작 시간보다 이르면, 정식 업무 시작 시간을 기준으로 삼는다.

```
effective_start = max(출근시간, 10:00:00)
```

| 출근 시간 | 유효 출근 시간 (effective_start) | 설명 |
|-----------|-------------------------------|------|
| 08:30 | 10:00 | 일찍 왔어도 10시부터 계산 |
| 09:55 | 10:00 | 일찍 왔어도 10시부터 계산 |
| 10:00 | 10:00 | 정시 출근 |
| 10:05 | 10:05 | 지각 — 실제 출근 시간 기준 |
| 10:30 | 10:30 | 지각 — 실제 출근 시간 기준 |

#### 야근 시작 시간

야근 시작 시간은 유효 출근 시간에 의무 근무시간(9시간)을 더한 값이다.

```
야근 시작 = effective_start + 9시간
```

| 출근 시간 | effective_start | 야근 시작 | 설명 |
|-----------|----------------|-----------|------|
| 09:00 | 10:00 | 19:00 | 일찍 왔어도 19시부터 야근 |
| 09:55 | 10:00 | 19:00 | 일찍 왔어도 19시부터 야근 |
| 10:00 | 10:00 | 19:00 | 정시 출근 → 19시부터 야근 |
| 10:05 | 10:05 | 19:05 | 5분 늦게 → 5분 밀림 |
| 10:30 | 10:30 | 19:30 | 30분 늦게 → 30분 밀림 |

#### 야근 인정 시간

야근 인정 기준은 유효 출근 시간에 12시간(9시간 의무 + 3시간 야근)을 더한 값이다.

```
야근 인정 시간 = effective_start + 12시간
```

#### 근무시간 계산

```python
effective_start = max(clock_in, clock_in.replace(hour=10, minute=0, second=0))
total_minutes = (clock_out - clock_in)           # 총 체류시간 (원본)
work_minutes  = (clock_out - effective_start) - 60분  # 근무시간 (점심 제외)
regular_minutes = min(work_minutes, 480분)       # 정규 근무시간 (최대 8시간)
overtime_minutes = max(0, work_minutes - 480분)  # 야근시간
```

### 4.2 퇴근 처리

#### 조기 퇴근 확인 다이얼로그

퇴근 버튼 클릭 시 의무 근무시간(9시간) 충족 여부에 따라 동작이 달라진다.

| 조건 | 동작 |
|------|------|
| 현재시각 < effective_start + 9시간 | 확인 다이얼로그 표시 후 사용자 선택에 따라 처리 |
| 현재시각 >= effective_start + 9시간 | 야근 결과물 업로드 다이얼로그 표시 후 퇴근 처리 |

**확인 다이얼로그 메시지 예시**

```
아직 의무 근무시간(9시간)이 채워지지 않았습니다.
퇴근 가능 시간까지 N시간 N분 N초 남았습니다.

어떤 방식으로 처리할까요?

[일반 퇴근]  [취소]
```

- **일반 퇴근** 선택 시: 실제 클릭 시각으로 퇴근 처리 진행
- **취소** 선택 시: 퇴근 처리 없이 닫기

### 4.3 정정 신청

- 출근 시간 클릭 시 정정 신청 다이얼로그 표시
- 희망 시간과 정정 사유 입력 필수
- `출근 / 정정신청` 요청 레코드 생성
- 퇴근은 정정신청을 사용하지 않고 `정상`과 `취소`만 사용한다.

### 4.4 퇴근 취소

퇴근은 정정신청, 승인, 확정, 반려를 사용하지 않는다. 잘못된 퇴근 레코드는 취소 레코드를 남겨 무효화한다.

1. 실제 퇴근 시각으로 `퇴근 / 정상` 레코드를 생성한다.
2. 퇴근을 취소하면 `퇴근 / 취소` 레코드를 생성하고, `sg_parent`는 취소 대상 `퇴근 / 정상` 레코드를 가리킨다.
3. 같은 퇴근 레코드에 대한 취소 레코드가 있으면 해당 퇴근 레코드는 최종 퇴근값 계산에서 제외한다.

```
A0: 퇴근 (17:03)
 │
 └─ C1: 퇴근 취소
     - sg_status = 취소
     - sg_parent = A0
```

### 4.5 출근 정정 승인/반려/포기

- 승인담당자만 처리 가능
- 승인 시: `출근 / 승인` 레코드 생성 → `출근 / 확정` 레코드 생성
- 반려 시: 반려 레코드 생성 (사유 필수)
- 반려 후 재신청 가능 (횟수 제한 없음)
- 반려 후 더 이상 진행하지 않으면 `출근 / 포기`로 종료한다.
- 승인/반려 레코드의 `sg_user`는 신청자로 유지하고, 처리자는 `sg_approver`에 기록한다.

### 4.6 식사 포기 정책

석식과 중식은 식사 신청 레코드를 만들지 않는다. 시스템은 기본적으로 사용자가 식사한 것으로 간주하고, 식사를 하지 않는 경우에만 포기 레코드를 남긴다.

#### 석식

- `석식 / 포기` 레코드가 있으면 저녁식사 안함으로 판단한다.
- `석식 / 포기` 레코드가 없으면 석식한 것으로 판단한다.
- 저녁식사 안함 체크박스를 켜면 `석식 / 포기` 레코드를 생성한다.
- 저녁식사 안함 체크박스를 끄면 해당 `석식 / 포기` 레코드를 삭제한다.
- `석식 / 정상`은 사용하지 않는다.

#### 중식

- `중식 / 포기` 레코드가 있으면 점심 안함으로 판단한다.
- `중식 / 포기` 레코드가 없으면 중식한 것으로 판단한다.
- 기존 `점심스킵` 데이터는 `중식 / 포기`로 마이그레이션한다.

### 4.7 야근 신청 워크플로우

야근은 출근 정정과 같은 `CustomNonProjectEntity10`을 사용하지만, 야근 전용 lifecycle 상태를 사용한다.

#### 최초 신청 → 승인 → 확정

1. 작업자가 야근을 신청하면 `야근 / 신청` 레코드를 생성한다.
2. 승인담당자가 승인하면 `야근 / 승인` 레코드를 생성한다.
3. 같은 요청에 대해 `야근 / 확정` 레코드를 추가 생성한다.
4. UI/API 상태는 기존 호환성을 위해 `확정`을 `approved` 상태로 취급한다.

```
R1: 야근 신청
 │  - sg_type = 야근
 │  - sg_status = 신청
 │  - sg_user = 작업자
 │  - sg_approver = 승인담당자
 │  - sg_time = 예상 퇴근 시간
 │
 ├─ D1: 야근 승인
 │   - sg_status = 승인
 │   - sg_parent = R1
 │   - sg_user = 작업자
 │   - sg_approver = 승인담당자
 │
 └─ C1: 야근 확정
     - sg_status = 확정
     - sg_parent = R1
     - sg_user = 작업자
     - sg_approver = 승인담당자
```

#### 신청 → 반려 → 재신청 → 승인

1. 승인담당자가 반려하면 `야근 / 반려` 레코드를 생성한다.
2. 작업자가 다시 신청하면 `야근 / 신청` 레코드를 생성한다.
3. 재신청 레코드의 `sg_parent`는 직전 반려된 야근 신청 레코드를 가리킨다.
4. 재신청이 승인되면 `야근 / 승인`과 `야근 / 확정` 레코드를 생성한다.

```
R1: 야근 신청
 │
 └─ D1: 야근 반려
     - sg_status = 반려
     - sg_parent = R1
     - sg_reason = 반려 사유
     │
     └─ R2: 야근 재신청
         - sg_status = 신청
         - sg_parent = R1
```

#### 신청 포기

- 승인 전, 승인 후, 반려 후 작업자가 야근 신청을 포기하면 `야근 / 포기` 레코드를 생성한다.
- 새 레코드는 `포기`를 사용하고, 기존 데이터의 `취소`는 호환 목적으로 terminal 상태로 함께 인정한다.
- Python API 상태 문자열은 기존 UI 호환을 위해 `canceled`를 유지한다.

### 4.8 승인 체계

#### 조직 구조와 ShotGrid 필드 매핑

| 직급 | ShotGrid 필드 | 설명 |
|------|--------------|------|
| 팀원 | `sg_part_supervisor` = 실장 | 실장이 출근 정정 또는 야근 신청을 승인 |
| 팀장 | `sg_part_supervisor` = 실장 | 실장이 출근 정정 또는 야근 신청을 승인 |
| 실장 | `sg_part_supervisor` = 비어있음 | 기본 승인담당자(`DEFAULT_CORRECTION_APPROVER`)가 출근 정정 또는 야근 신청을 승인 |

#### 승인 권한 판별

출근 정정 승인 대기 뱃지 표시 여부는 **사전 판별 없이** `Attendance.get_pending_corrections()`를 호출하여 결과가 있으면 표시한다. 야근 승인 대기는 `Attendance.get_pending_overtime_requests()`로 별도 조회한다. department와 무관하게, 요청자의 `sg_part_supervisor`에 등록된 사람이 승인할 수 있으며, `sg_part_supervisor`가 비어 있으면 기본 승인담당자(`DEFAULT_CORRECTION_APPROVER`)만 승인할 수 있다.

#### 출근 정정 신청 조회 범위 (누구의 정정을 볼 수 있는가)

`Attendance.get_pending_corrections(approver)` 메서드의 필터링 기준:

1. `sg_type = '출근'`, `sg_status = '정정신청'`인 레코드 전체 조회
2. `sg_approver`가 있으면 해당 값을 승인담당자로 사용한다.
3. `sg_approver`가 비어 있는 기존 레코드는 **요청자의 `sg_part_supervisor`가 있으면 그 첫 번째 값**을 사용하고, 비어 있으면 **기본 승인담당자(`DEFAULT_CORRECTION_APPROVER`)를 승인담당자로 사용**한다.
4. 계산된 승인담당자가 현재 승인자와 같고 **본인 건이 아닌 것**만 필터링한다.
5. 이미 승인/반려/포기된 요청은 제외

```python
# sg_approver를 우선 사용하고, 기존 레코드는 요청자의 sg_part_supervisor를 fallback으로 사용한다.
correction_approver = item.get('sg_approver')
if not correction_approver:
    supervisors = item.get('sg_user.HumanUser.sg_part_supervisor')
    correction_approver = Attendance._get_correction_approver_from_supervisors(supervisors)
if correction_approver and correction_approver.get('id') == approver_id:
    filtered.append(item)
```

#### 승인 흐름 요약

| 요청자 | 승인자 | 판별 기준 |
|--------|--------|----------|
| 팀원 | 요청자의 `sg_part_supervisor` | 요청자의 실장이 승인 |
| 팀장 | 요청자의 `sg_part_supervisor` | 요청자의 실장이 승인 |
| 실장 | `DEFAULT_CORRECTION_APPROVER` | `sg_part_supervisor` 비어있음 → 기본 승인담당자가 승인 |

### 4.9 불변성 규칙

- **모든 레코드는 생성 후 수정하지 않음**
- 상태 변경은 새 레코드 생성으로 표현
- 감사 추적이 데이터 구조 자체로 보장됨

## 5. 무결성 규칙

### 5.1 필수 검증

1. **타입별 상태 제한**: `sg_type`마다 허용된 `sg_status`만 생성할 수 있다.
2. **출근 정정 원본 존재**: `출근 / 정정신청`은 원본 `출근 / 정상` 레코드를 `sg_parent`로 가져야 한다.
3. **출근 승인/확정/반려/포기 대상 존재**: 출근 정정 처리 레코드는 처리 대상 요청 또는 반려 레코드를 `sg_parent`로 가져야 한다.
4. **퇴근 상태 제한**: 퇴근은 `정상`과 `취소`만 사용하며, `정정신청`, `승인`, `확정`, `반려`, `포기`를 사용하지 않는다.
5. **식사 기본값**: 석식과 중식은 사용자가 포기하지 않는 한 식사한 것으로 간주한다.
6. **식사 포기 상태 제한**: 석식과 중식은 `포기`만 사용한다. `정상`, `신청`, `취소` 등 다른 상태를 사용하지 않는다.
7. **식사 포기 해제**: 저녁식사 안함 또는 점심 안함 체크를 끄면 해당 포기 레코드를 삭제한다.
8. **야근 신청 상태**: 야근은 `신청`, `승인`, `확정`, `반려`, `포기`, `취소`만 사용하며 `정정신청`을 사용하지 않는다.
9. **야근 재신청 parent**: 반려 후 다시 신청한 `야근 / 신청`의 `sg_parent`는 직전 반려된 야근 신청 레코드를 가리킨다.
10. **야근 terminal 상태**: `확정`, `반려`, `포기`, 기존 `취소`는 해당 야근 요청을 처리 완료 상태로 만든다.
11. **최종 출근값**: 같은 `sg_user + sg_date + sg_type=출근` 조합에 `정상` 또는 `확정` 레코드가 여러 개 있을 수 있으며, 가장 나중에 생성된 확정 계열 레코드를 최종 출근값으로 사용한다.
12. **최종 퇴근값**: 같은 `sg_user + sg_date + sg_type=퇴근` 조합의 `정상` 레코드 중 `취소` 레코드의 `sg_parent`로 참조되지 않은 최신 레코드를 최종 퇴근값으로 사용한다.

### 5.2 마이그레이션 규칙

| 기존 데이터 | 처리 |
|-------------|------|
| `기타 / 정상 / sg_reason=석식포기` | `석식 / 포기`로 마이그레이션 |
| `기타 / 정상 / sg_reason=석식` | 삭제. 레코드가 없으면 석식한 것으로 간주하므로 대체 레코드를 만들지 않는다. |
| `석식 / 정상` | 삭제. 레코드가 없으면 석식한 것으로 간주하므로 대체 레코드를 만들지 않는다. |
| `기타 / 정상 / sg_reason=점심스킵` | `중식 / 포기`로 마이그레이션 |
| `야근 / 정정신청` | `야근 / 신청`으로 마이그레이션 |
| 그 외 `기타` | 운영 데이터 확인 후 별도 정책 결정 |

## 6. 설계 결정 (Design Decisions)

### 6.1 sg_date + sg_time 중복 허용

**문제**: `sg_time`(DateTime)에 이미 날짜가 포함되어 있어 `sg_date`(Date)와 중복됨.

**결정**: 의도적 비정규화로 유지

**이유**:
- 야근 시 날짜 불일치가 정상 케이스 (3/6 출근 → 3/7 01:30 퇴근)
- `sg_date`는 논리적 근무일 (비즈니스 키), `sg_time`은 물리적 시간
- 쿼리 편의성 (`sg_date`로 필터링)
- 무결성은 코드에서 보장

### 6.2 완전 불변 이벤트 로그 방식

**문제**: 출퇴근 기록을 어떻게 관리할 것인가?

**결정**: 레코드는 생성만 하고, 절대 수정하지 않음

**이유**:
- 감사 추적이 데이터 구조 자체로 보장됨
- 모든 변경 이력이 자동으로 보존됨
- 데이터 위변조 불가능
- 디버깅 및 문제 추적 용이

### 6.3 sg_user / sg_approver 필드 분리

**문제**: 대상 직원과 승인자를 별도 필드로 관리할 것인가?

**결정**: `sg_user`는 대상 직원/신청자로 고정하고, 승인담당자 또는 실제 처리자는 `sg_approver`에 기록

**이유**:
- ShotGrid 화면에서 신청자와 승인담당자를 즉시 구분할 수 있음
- 확정/요청/승인/반려 레코드 모두 `sg_user`를 대상 직원 기준으로 조회할 수 있음
- 승인 대기, 승인, 반려, 취소 처리자는 `sg_approver`로 일관되게 추적 가능
- 기존 `sg_parent` 체인은 이벤트 관계 추적에만 집중할 수 있음

### 6.4 sg_parent 필드명 선택

**문제**: 연결 대상 레코드를 뭐라고 부를 것인가?

**검토한 후보**:
- `sg_related_attendance` — 너무 김
- `sg_upstream` / `sg_downstream` — 방향 햷갈림 (데이터 흐름 vs 이력 추적 방향)
- `sg_parent` — 계층 구조 명확, 직관적

**결정**: `sg_parent` 사용

**이유**:
- "이 레코드의 Parent는 뭐야?"가 직관적
- upstream/downstream보다 비즈니스 용어로 자연스러움
- 코드가 깔끔해짐 (`request_log['sg_parent']`)

## 7. 유급휴가 엔티티

유급휴가(연차, 반차 등)는 출퇴근 기록과 별도로 관리하며, **단순 기록용**으로 설계한다.

### 7.1 설계 방식

- **별도 엔티티**: 출퇴근 기록(`CustomNonProjectEntity10`)과 분리
- **날짜별 개별 레코드**: 하루에 하나의 휴가 레코드 (반차도 동일)
- **단순 기록**: 상태 변경(정정신청/승인/반려) 없음, 생성 후 수정/삭제 없음
- **사유 필수**: 휴가 신청 시 사유 기록

### 7.5 출퇴근 현황과의 관계

```
[출퇴근 현황 조회 시]
  │
  ├─ 출퇴근 기록 (CustomNonProjectEntity10) 조회
  │   - 출근/퇴근/석식/중식/야근
  │
  ├─ 휴가 기록 (CustomNonProjectEntity09) 조회
  │   - 유급휴가: 연차/반차/반반차
  │   - 무급휴가: 무급휴가(일반/병가/육아/기타), 병가, 경조, 휴직휴가
  │
  └─ 무급휴가 기록 (CustomNonProjectEntity01) 참조
      - 기간이 정해지지 않은 무급휴가 진행 중 여부 확인
      - 확정된 무급휴가는 휴가 엔티티로 동기화 완료
```

### 7.6 무급휴가 동기화 정책

무급휴가(`CustomNonProjectEntity01`)는 기간 단위로 관리되지만, 출퇴근 현황 조회의 편의성을 위해 **휴가 엔티티(`CustomNonProjectEntity09`)에 날짜별 레코드로 동기화**한다.

#### 동기화 방식

| 항목 | 정책 |
|------|------|
| **동기화 주체** | ValiDuck 스케줄러 (C:/dev/infx/validuck) |
| **동기화 주기** | 매일 오전 7시 |
| **입력 방식** | 당일 날짜만 생성 (과거/미래 한꺼번에 생성 금지) |
| **기간 변경 대응** | 기간이 줄어들면 이미 생성된 미래 레코드는 유지 (역사적 기록)<br>기간이 늘어나면 다음 날 7시에 새로 생성 |
| **종류 구분** | 무급휴가 엔티티의 `sg_leave_type` 값을 그대로 휴가 엔티티 `sg_type`으로 사용 |
| **중복 방지** | `sg_user + sg_date + sg_type` 조합으로 중복 체크 |
| **소스 오브 트루스** | 확정 후(익일)는 휴가 엔티티가 소스 오브 트루스 |

#### 아키텍처

```
무급휴가 엔티티(기간 관리) 
    ↓ (ValiDuck 매일 7시)
flova.timelog.attendance.sync_unpaid_leave_for_date()
    - 무급휴가 엔티티 조회
    - 당일 해당자 필터링
    - 중복 체크 (user+date+type)
    - 휴가 엔티티 생성
    ↓
휴가 엔티티(날짜별 레코드) ← 출퇴근 현황 조회 시 활용
```

## 8. 설계 결정 (Design Decisions)

### 8.5 유급휴가를 별도 엔티티로 분리

**문제**: 유급휴가를 어떻게 관리할 것인가?

**검토한 옵션**:

| 옵션 | 장점 | 단점 |
|------|------|------|
| **A. TimeLog 계속 사용** | 이미 구현됨 | 타임로그 본래 목적(작업시간)과 혼재 |
| **B. 출퇴근 기록에 추가** | 날짜별 레코드 존재 | 출퇴근(근무)과 휴가(비근무) 성격 상이 |
| **C. 무급휴가 엔티티에 포함** | 휴가 한 곳 관리 | 기간단위/날짜단위 구조 불일치 |
| **D. 별도 엔티티 생성** | 의미적 정합성, 확장성 | 조회 시 2회 API 호출 |

**결정**: **옵션 D (별도 엔티티)** 사용 — `CustomNonProjectEntity09`

**이유**:
- **의미적 정합성**: 타임로그는 "작업 시간", 휴가는 "근무하지 않는 시간"
- **데이터 구조 일관성**: 출퇴근 기록(`CustomNonProjectEntity10`)과 동일한 날짜별 이벤트 로그 패턴
- **성격 차이**: 무급휴가는 장기(기간 단위), 유급휴가는 단기(반차 등)
- **단순 기록**: 출퇴근과 달리 정정/승인/반려 워크플로우 불필요 (수정은 삭제 후 재생성)

### 8.6 유급휴가에 sg_status/sg_parent 미사용

**문제**: 유급휴가도 출퇴근처럼 정정 신청/승인/반려가 필요한가?

**결정**: **필드 제외** — `sg_status`, `sg_parent` 없음

**이유**:
- 유급휴가는 단순 기록용 (신청→확정 단순 흐름)
- 수정 필요 시: 삭제 후 재생성 (불변성 유지)
- 잔여 연차는 ShotGrid HumanUser 필드 또는 별도 관리

### 8.7 유급휴가에 sg_reason 필수

**결정**: `sg_reason` 필드 필수 (text)

**이유**:
- 휴가 사유 추적 및 통계 필요
- 연차/반차 구분과 별개로 구체적 사유 기록
